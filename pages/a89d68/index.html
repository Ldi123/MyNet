传统 Spring MVC 项目接口改造 CXF WebService 踩坑复盘

# 传统 Spring MVC 项目把Controller层定义的POST接口改成 CXF WebService 踩坑复盘

# 一、项目背景

  • 项目类型：传统 Spring MVC / SSM 项目（非 Spring Boot）
  • 需求：对外提供 WebService 接口，接收 XML 格式入参并返回 XML 响应
  • 核心框架：Spring + Spring MVC + MyBatis + CXF
  • 前提条件：已经有完整的以 POST 方式调用的接口，需要改成webService方式调用，主要的改造点是添加@WebService注解和@Component注解（也可以是@Service注解，是为了还能继续使用@Autowired），移除@Controller等Spring MVC注解。

# 二、整体踩坑历程与解决路线

# 1. 依赖错误：误用 Spring Boot Starter

问题表现

  • 引入 cxf-spring-boot-starter-jaxws
  • 找不到 Spring Boot 相关类、Bean 无法注入、启动报错

尝试方法

  • 更换 CXF 4.2.0 → 不兼容 Spring MVC
  • 更换 CXF 3.5.8 → 仍因 Starter 不匹配报错

正确方案

彻底删除 Starter，改用纯 Spring MVC 原生依赖：

<dependency>
    <groupId>org.apache.cxf</groupId>
    <artifactId>cxf-rt-frontend-jaxws</artifactId>
    <version>3.5.8</version>
</dependency>
<dependency>
    <groupId>org.apache.cxf</groupId>
    <artifactId>cxf-rt-transports-http</artifactId>
    <version>3.5.8</version>
</dependency>

# 2. 启动类写法错误：使用 Spring Boot 专用 API

问题表现

  • 使用 implements CommandLineRunner
  • 注入 @Autowired private Bus bus
  • 报错：NoSuchBeanDefinitionException: No bean named 'cxf' available

尝试方法

  • 手动构造 Bus → 失败
  • 保留 Spring Boot 风格配置 → 持续报错

正确方案

删除 CommandLineRunner，使用 @PostConstruct 在 Spring 容器启动后发布服务：

@Configuration
public class WebServiceConfig {
    @Autowired
    private MyTestController myTestController;

    @PostConstruct
    public void publishWebService() {
        JaxWsServerFactoryBean factory = new JaxWsServerFactoryBean();
        factory.setServiceBean(myTestController);
        factory.setAddress("/MyTestService");
        factory.create();
    }
}

# 3. 缺少 CXFServlet 导致接口 404

问题表现

  • 项目启动无报错
  • 访问接口直接 404

尝试方法

  • 直接访问 /MyTestService
  • 去掉项目名访问 → 依旧 404

正确方案

web.xml 追加独立 Servlet 配置（不影响原有 DispatcherServlet）：

<servlet>
    <servlet-name>cxf</servlet-name>
    <servlet-class>org.apache.cxf.transport.servlet.CXFServlet</servlet-class>
</servlet>
<servlet-mapping>
    <servlet-name>cxf</servlet-name>
    <url-pattern>/ws/*</url-pattern>
</servlet-mapping>

# 4. 缺少 cxf.xml 导入导致 Bean 缺失

问题表现

No bean named 'cxf' available
1

尝试方法

  • 手动定义 cxf Bean → 失败
  • 调整配置文件加载顺序 → 无效

正确方案

在 Spring 根配置文件（如 spring-service-bean.xml）中导入：

<import resource="classpath:META-INF/cxf/cxf.xml" />

# 5. XML 解析报错：不允许有匹配 "[xX][mM][lL]" 的处理指令目标

问题表现

  • 入参为 SOAP 协议
  • CDATA 内部又嵌套了 <?xml version="1.0" encoding="gbk"?>
  • XML 声明前存在换行、空格、不可见字符

尝试方法

  • 调整编码 UTF-8 / GBK → 无效
  • 只 trim () → 仍偶现报错

最终稳定方案

// 1. 删除所有 XML 声明
xmlContent = xmlContent.replaceAll("<\\?xml.*?\\?>", "");
// 2. 清除首尾空白、换行、不可见字符
xmlContent = xmlContent.trim();

# 三、关键经验教训

# 1. 分清项目类型：Spring Boot vs 传统 Spring MVC

  • Spring Boot：使用 cxf-spring-boot-starter-jaxws
  • 传统 SSM：必须使用原生 cxf-rt-* 依赖
  • 禁止混用 Spring Boot 相关 API（ServletRegistrationBean、CommandLineRunner 等）

# 2. CXF 在传统项目中必须配置两点

  • web.xml 中注册独立 CXFServlet
  • Spring 配置中导入 cxf.xml

# 3. WebService 接口地址规则

  • 格式：/项目名/ servletPath /服务名?wsdl
  • 不可直接访问服务名，必须走 Servlet 前缀

# 4. XML 嵌套是高频报错点

  • SOAP envelope 本身已是 XML
  • 内部业务 XML 必须放在 CDATA 或做转义
  • 解析前务必删除内部 XML 声明、清理空白字符

# 5. XML 解析器极度严格

  • XML 声明必须在第一位
  • 前面不能有任何字符（换行、空格、BOM 都不行）
  • 不允许出现多个 XML 声明

# 四、正确可直接复用的最小集成方案

  1. 引入正确 CXF 依赖
  2. web.xml 配置 CXFServlet
  3. Spring 配置导入 cxf.xml
  4. 使用 @PostConstruct 发布服务
  5. 解析入参时清理 XML 声明与空白字符
  6. 返回值做 XML 转义，避免嵌套格式错误

# 五、总结

本次踩坑核心原因是混用 Spring Boot 与传统 Spring MVC 用法，导致依赖、配置、启动方式全线不匹配。

只要严格按照传统 SSM 项目规范集成 CXF，整体流程非常稳定，接口可正常发布、调用、解析、响应，且对原有业务代码零侵入、零影响。
